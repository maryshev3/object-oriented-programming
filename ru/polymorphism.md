# Полиморфизм

## Введение

Представьте, что мы разрабатываем сервис для работы с базами данных. В проекте решено поддерживать два типа СУБД: PostgreSQL и Microsoft SQL Server. Для каждого из них мы написали свой класс со своим способом подключения и выполнения запросов.

```cpp
#include <iostream>
#include <string>

class PostgresConnection {
    public:
        void connect(const std::string& connection_string) {
            std::cout << "[PostgreSQL] Подключаемся к " << connection_string << "..." << std::endl;
            // Специфичная логика подключения для PostgreSQL
        }

        std::string execute_query(const std::string& sql) {
            std::cout << "[PostgreSQL] Выполняем запрос: " << sql << std::endl;
            // Обработка запроса средствами PostgreSQL
            return "PostgreSQL result";
        }

        void disconnect() {
            std::cout << "[PostgreSQL] Отключаемся." << std::endl;
        }
};
```

```cpp
#include <iostream>
#include <string>

class MssqlConnection {
    public:
        void connect(const std::string& connection_string) {
            std::cout << "[MSSQL] Подключаемся к " << connection_string << "..." << std::endl;
            // Специфичная логика подключения для Microsoft SQL Server
        }

        std::string execute_query(const std::string& sql) {
            std::cout << "[MSSQL] Выполняем запрос: " << sql << std::endl;
            // Обработка запроса средствами MSSQL
            return "MSSQL result";
        }

        void disconnect() {
            std::cout << "[MSSQL] Отключаемся." << std::endl;
        }
};
```

Теперь нам нужно написать функцию, которая принимает подключение к любой базе данных и выполняет тестовый запрос. Сразу приходит в голову написать несколько перегруженных функций — по одной для каждого типа подключения:

```cpp
void run_test_query(PostgresConnection& conn) {
    conn.connect("host=localhost dbname=test");
    std::string result = conn.execute_query("SELECT 1");
    std::cout << "Результат: " << result << std::endl;
    conn.disconnect();
}

void run_test_query(MssqlConnection& conn) {
    conn.connect("Server=localhost;Database=test");
    std::string result = conn.execute_query("SELECT 1");
    std::cout << "Результат: " << result << std::endl;
    conn.disconnect();
}
```

Проблемы такого подхода очевидны:

- ***дублирование кода***: тело обеих функций практически одинаково, меняются только строки подключения. Если потребуется изменить логику (например, добавить обработку ошибок), придётся править все перегрузки;

- ***сложность расширения***: при добавлении поддержки ещё одной СУБД (скажем, Oracle) потребуется написать третью перегрузку. Количество версий функции растёт с каждым новым типом подключения;

- ***жёсткая привязка***: вызывающий код должен знать конкретный тип объекта и явно выбирать нужную перегрузку.

Хотелось бы написать одну функцию *run_test_query*, которая работает с любым подключением и не требует изменений при появлении новых типов баз данных. Для этого все классы подключений должны предоставлять общий интерфейс — одинаковый набор методов. Тогда *run_test_query* сможет принимать объект через ссылку на этот общий интерфейс и вызывать *connect*, *execute_query*, *disconnect*, не задумываясь о реальном типе.

Именно эту возможность и даёт полиморфизм: во время исполнения программы вызывается та версия метода, которая соответствует фактическому типу переданного объекта, даже если мы обращаемся к нему через ссылку (или указатель) базового класса.

<div class="task-box"> <p><strong>Полиморфизм</strong> — один из ключевых принципов объектно-ориентированного программирования, позволяющий объектам с одинаковым интерфейсом (базовым классом) иметь различную реализацию поведения.</p> </div>

Простыми словами, полиморфизм отвечает на вопрос: "Как сделать так, чтобы код, работающий с базовым классом, автоматически применял правильную реализацию для каждого конкретного наследника?"

В C++ полиморфизм времени выполнения реализуется с помощью ***виртуальных функций*** и ***абстрактных классов***.

## Применение в разработке

Переход к термину "полиморфизм" происходит на этапе проектирования, когда мы обнаруживаем группу классов, выполняющих одни и те же операции, но разными способами. Классические примеры:

- ***драйверы баз данных***. Все соединения с СУБД поддерживают методы *connect*, *executeQuery*, *disconnect*. Это позволяет писать прикладной код, независимый от конкретной базы данных;

- ***графические редакторы***. Все фигуры (круги, прямоугольники, треугольники) наследуются от класса ***Shape*** с виртуальным методом *draw()*. Основной сервис для отрисовки всех фигур на полотне вызывает метод *draw()* через ссылку на базовый класс;

- ***игровые движки***. Игровые объекты наследуются от ***GameObject*** с методами *update()*, *render()*. При формировании каждого нового кадра для всех объектов вызываются данные методы без явного указания реального типа для отображения на сцене;

- ***платёжные системы***. Для реализации платёжных шлюзов как правило используют базовый класс ***BankProcessor*** с виртуальным методом *processPayment(amount)*. Конретным банковским провайдерам соответствуют реализации по типу ***PayPalProcessor***, ***StripeProcessor***, ***SberbankProcessor***. Интернет-магазин работает с интерфейсом и не зависит от конкретного платёжного шлюза.

Таким образом, полиморфизм применяется всегда, когда мы хотим писать код, отвязанный от конкретных классов, и при этом иметь возможность подставлять различные реализации без изменения этого кода.

## Полиморфизм в C++

### Виртуальные методы

Чтобы полиморфизм заработал, в C++ вводится понятие виртуальной функции. Обычный (невиртуальный) метод класса привязывается к вызову на этапе компиляции — компилятор смотрит на указанный тип ссылки или указателя и вызывает метод именно этого типа.

Если мы хотим, чтобы решение о том, какую версию метода вызывать, принималось во время выполнения на основе реального типа объекта, метод необходимо объявить с ключевым словом virtual в базовом классе.

```cpp
class DatabaseConnection {
    public:
        virtual void connect(const std::string& conn_str) {
            // Базовая реализация (может быть пустой или общей)
        }
};
```

Когда метод объявлен виртуальным, компилятор строит для класса ***таблицу виртуальных функций (vtable)***, в которой хранятся указатели на фактические реализации. При вызове метода через указатель или ссылку на базовый класс программа заглядывает в эту таблицу и вызывает версию, соответствующую истинному динамическому типу объекта. Это называется ***поздним связыванием*** или ***динамическим связыванием***.

В наследниках виртуальную функцию можно переопределить. Для явного выражения намерения и защиты от ошибок используется ключевое слово ***override***:

```cpp
class PostgresConnection : public DatabaseConnection {
    public:
        void connect(const std::string& conn_str) override {
            // специфика PostgreSQL
        }
};
```

Если сигнатура метода не совпадает с виртуальной функцией базового класса, компилятор выдаст ошибку.

### Абстрактные классы

Зачастую базовый класс задаёт только интерфейс и не имеет осмысленной реализации некоторых методов. Например, метод *execute_query* не может быть написан в общем виде, потому что каждая база данных выполняет запрос по-своему. Чтобы запретить создание экземпляров такого «недоделанного» класса и обязать наследников предоставить свои реализации, используется ***чисто виртуальная функция***. Она объявляется с = 0 в конце:

```cpp
virtual std::string execute_query(const std::string& sql) = 0;
```

Класс, содержащий хотя бы одну чисто виртуальную функцию, становится ***абстрактным***. Создать его экземпляр напрямую нельзя — только экземпляры наследников, реализовавших все чисто виртуальные функции. Абстрактный класс выступает в роли ***интерфейса***: он говорит, что должен уметь любой наследник, но не диктует, как именно.

### Пример использования абстрактного класса

Перепишем пример с базами данных, используя полиморфизм.

Сначала определим общий абстрактный интерфейс ***DatabaseConnection***:

```cpp
#include <iostream>
#include <string>

class DatabaseConnection {
    public:
        // Виртуальный деструктор, для правильного вызова деструктора наследников.
        virtual ~DatabaseConnection() {}

        // Основные методы, общие для всех реализаций. Фактически задают интерфейс.
        virtual void connect(const std::string& connection_string) = 0;
        virtual std::string execute_query(const std::string& sql) = 0;
        virtual void disconnect() = 0;
};
```

Класс является абстрактным, т.к. все три метода чисто виртуальные. Любое подключение обязано их реализовать.

Конкретные наследники для PostgreSQL и MSSQL:

```cpp
class PostgresConnection : public DatabaseConnection {
    public:
        void connect(const std::string& connection_string) override {
            std::cout << "[PostgreSQL] Подключаемся к " << connection_string << std::endl;
        }

        std::string execute_query(const std::string& sql) override {
            std::cout << "[PostgreSQL] Выполняем: " << sql << std::endl;
            // Выполнение запроса и возврат результата
            return "PostgreSQL_result";
        }

        void disconnect() override {
            std::cout << "[PostgreSQL] Отключаемся." << std::endl;
        }
};

class MssqlConnection : public DatabaseConnection {
public:
    void connect(const std::string& connection_string) override {
        std::cout << "[MSSQL] Подключаемся к " << connection_string << std::endl;
    }

    std::string execute_query(const std::string& sql) override {
        std::cout << "[MSSQL] Выполняем: " << sql << std::endl;
        return "MSSQL_result";
    }

    void disconnect() override {
        std::cout << "[MSSQL] Отключаемся." << std::endl;
    }
};
```

Теперь мы можем написать одну функцию для выполнения запроса, ничего не знающую о конкретном типе базы:

```cpp
void run_query(DatabaseConnection& db, const std::string& connection_str, const std::string& sql) {
    db.connect(connection_str);
    std::string result = db.execute_query(sql);
    std::cout << "Получен результат: " << result << std::endl;
    db.disconnect();
}
```

Вся магия в том, что вызовы *db.connect()*, *db.execute_query()*, *db.disconnect()* внутри *run_query* — полиморфные. Если передан объект ***PostgresConnection***, то отработают методы PostgreSQL; если ***MssqlConnection*** — методы MSSQL. Сама функция *run_query* этого не проверяет и не знает.

Пример использования:

```cpp
int main() {
    PostgresConnection pg;
    MssqlConnection ms;

    run_query(pg, "host=localhost dbname=mydb", "SELECT * FROM users;");
    std::cout << "-------------------\n";
    run_query(ms, "Server=localhost;Database=mydb", "SELECT * FROM orders;");

    return 0;
}
```

Вывод:

```
[PostgreSQL] Подключаемся к host=localhost dbname=mydb
[PostgreSQL] Выполняем: SELECT * FROM users;
Получен результат: PostgreSQL_result
[PostgreSQL] Отключаемся.
-------------------
[MSSQL] Подключаемся к Server=localhost;Database=mydb
[MSSQL] Выполняем: SELECT * FROM orders;
Получен результат: MSSQL_result
[MSSQL] Отключаемся.
```

Если в будущем мы добавим ***OracleConnection***, функция *run_query* останется без изменений. Достаточно унаследовать новый класс от ***DatabaseConnection*** и реализовать три чисто виртуальных метода.

<div class="task-box"> <p><strong>Важное замечание:</strong> Полиморфное поведение работает только при обращении к объекту через ссылку или указатель на базовый класс. Если передать объект по значению (например, в функцию, принимающую <code>DatabaseConnection</code>), произойдёт срезка — скопируется только базовая часть, а информация о производном типе потеряется, и вызовы методов пойдут по базовому классу.</p> </div>

## Рекомендации по оформлению кода

При использовании полиморфизма в C++ следует придерживаться следующих правил:

- ***всегда используйте override при переопределении***. Ключевое слово override заставляет компилятор проверить, что вы действительно переопределяете виртуальную функцию базового класса, а не создаёте новую;

- ***предпочитайте ссылки указателям, где это возможно***. Ссылка не может быть нулевой и делает интерфейс более безопасным;

- ***избегайте явного приведения типов (dynamic_cast)***. Если вам постоянно приходится выяснять реальный тип объекта, значит, интерфейс базового класса спроектирован недостаточно полно. Хороший полиморфный код обходится вызовами виртуальных функций без проверок типа;

- ***базовый интерфейс должен быть минимальным и осмысленным***. Включайте в абстрактный класс только те методы, которые действительно являются общими для всех наследников. Не раздувайте интерфейс "на всякий случай".
