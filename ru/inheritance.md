# Наследование

## От дублирования кода к общей основе

Представьте, что мы разрабатываем банковскую систему. В какой-то момент возникает потребность создать два типа счетов: обычный сберегательный и чековый (расчётный). Оба счёта имеют много общего: номер счёта, имя владельца, баланс, возможность пополнения и снятия средств. Если реализовывать их независимо, код получится примерно таким (фрагменты):

```cpp
class SavingsAccount {
    public:
        SavingsAccount(const std::string& owner, double initial);
        void deposit(double amount);
        bool withdraw(double amount);  // нельзя уйти в минус
        void display() const;
    private:
        std::string m_owner;
        double m_balance;
        double m_interest_rate;  // специфическое поле
};

class CheckingAccount {
    public:
        CheckingAccount(const std::string& owner, double initial);
        void deposit(double amount);
        bool withdraw(double amount);  // можно уходить в небольшой минус (овердрафт)
        void display() const;
    private:
        std::string m_owner;
        double m_balance;
        double m_overdraft_limit;  // специфическое поле
};
```

Сразу видны проблемы: поля *m_owner*, *m_balance* и методы *deposit*, *withdraw* продублированы. Если в логику пополнения потребуется внести изменение, придётся править оба класса. Кроме того, такой код плохо отражает наше понимание предметной области: на самом деле и SavingsAccount, и CheckingAccount — это разновидности банковского счёта, у них есть общая природа.

Хотелось бы вынести всё общее в один класс ***Account***, а специфические детали оставить в производных классах. Тогда можно было бы сказать: "Сберегательный счёт — это банковский счёт, который дополнительно имеет процентную ставку и запрещает уходить в минус", а "Чековый счёт — это банковский счёт, который дополнительно имеет лимит овердрафта". Вынесение общей части и создание на её основе специализированных версий — и есть суть наследования.

Таким образом, наследование позволяет нам выстроить иерархию «общее -> частное», избавляясь от дублирования и делая программу более понятной.

<div class="task-box"> <p><strong>Наследование</strong> — это механизм объектно-ориентированного программирования, позволяющий создавать новый класс на основе существующего. Новый класс (производный, наследник) получает все поля и методы базового класса (родителя) и может добавлять собственные либо изменять унаследованное поведение. Наследование моделирует отношение "является": наследник является разновидностью базового класса.</p> </div>

## Когда переходят к наследованию

В реальной разработке идея наследования возникает не на пустом месте. Как только вы начинаете проектировать систему и замечаете, что несколько сущностей имеют общие атрибуты и поведение, пора задуматься о выделении базового класса. Классические примеры:

- ***Пользовательский интерфейс:*** все элементы интерфейса (кнопки, текстовые поля, списки) наследуют от класса ***Widget***, который определяет общие методы *draw()*, *handle_event()*, *set_position()*. Это позволяет обрабатывать множество разных элементов единообразно;

- ***Игровые движки:*** базовый класс ***GameObject*** содержит координаты, скорость, спрайт. От него наследуются ***Player***, ***Enemy***, ***Bullet***, добавляя своё поведение;

- ***Обработка исключений:*** в стандартной библиотеке C++ все исключения наследуют от ***std::exception***, что даёт общий интерфейс *what()*;

- ***Работа с файлами и потоками:*** в стандартной библиотеке C++ все потоки на ввод наследуюися от ***std::istream***. Например: ***std::ifstream*** (файловый ввод), ***std::istringstream*** (строковый). Базовый класс предоставляет общий интерфейс операций ввода, но реализации отличаются.

Переход к наследованию обычно оформляется на этапе проектирования, когда вы анализируете выделенные сущности и решаете: "Это разные проявления одного и того же?" Если ответ "да" и у классов есть общие поля/методы, вы выносите их в базовый класс, а в наследниках реализуете только отличия. При этом важно следовать правилу: ***наследование должно отражать отношение "является"***. Например, сберегательный счёт является банковским счётом — наследование оправдано. Если же связь скорее "содержит" или "использует", предпочтение отдаётся композиции.

## Наследование в C++

Рассмотрим, как та же банковская система реализуется с помощью наследования. Сначала определим базовый класс ***Account***, содержащий общую функциональность:

```cpp
#include <string>
#include <iostream>

class Account {
    public:
        Account(const std::string& owner, double initial)
            : m_owner(owner), m_balance(initial) {}

        void deposit(double amount) {
            m_balance += amount;
        }

        // Виртуальный метод: в базовом классе задаёт общую логику,
        // но наследники могут её переопределить.
        virtual bool withdraw(double amount) {
            if (amount <= m_balance) {
                m_balance -= amount;
                return true;
            }
            return false;
        }

        virtual void display() const {
            std::cout << "Владелец: " << m_owner << ", баланс: " << m_balance;
        }

        virtual ~Account() {}  // виртуальный деструктор. Его могут переопределить наследники, т.к. у них могут быть собственные поля для высвобождения памяти.

    protected:
        std::string m_owner;
        double m_balance;
};
```

Теперь создадим наследника — класс ***SavingsAccount***. Для наследования используется синтаксис ***class Имя : спецификатор_доступа БазовыйКласс***. Спецификатор public означает, что все публичные члены базового класса останутся публичными и у наследника (наиболее частый вариант).

<div class="task-box"> <p>Во всех современных языках программирования когда упоминается насследование, имеется ввиду наследование со спецификатором public. Существуют также спецификаторы protected, private, но их использовать не рекомендуется.</p> </div>

```cpp
class SavingsAccount : public Account {
    public:
        SavingsAccount(const std::string& owner, double initial, double rate)
            : Account(owner, initial),  // вызов конструктора базового класса
            m_interest_rate(rate) {}

        // Переопределяем withdraw с дополнительным условием: баланс не может стать отрицательным
        bool withdraw(double amount) override {
            if (amount <= m_balance) {
                m_balance -= amount;
                return true;
            }
            return false;
        }

        void display() const override {
            std::cout << "[Сберегательный] ";
            Account::display();  // используем базовую реализацию
            std::cout << ", ставка: " << m_interest_rate << "%\n";
        }

    private:
        double m_interest_rate;
};
```

Аналогично, класс ***CheckingAccount***:

```cpp
class CheckingAccount : public Account {
    public:
        CheckingAccount(const std::string& owner, double initial, double limit)
            : Account(owner, initial), m_overdraft_limit(limit) {}

        // Переопределённый withdraw разрешает уходить в минус в пределах лимита
        bool withdraw(double amount) override {
            if (m_balance + m_overdraft_limit >= amount) {
                m_balance -= amount;  // баланс может стать отрицательным
                return true;
            }
            return false;
        }

        void display() const override {
            std::cout << "[Чековый] ";
            Account::display();
            std::cout << ", лимит овердрафта: " << m_overdraft_limit << "\n";
        }

    private:
        double m_overdraft_limit;
};
```

Использование:

```cpp
int main() {
    SavingsAccount sav("Alice", 1000.0, 5.0);
    CheckingAccount chk("Bob", 500.0, 200.0);

    sav.deposit(200);
    chk.withdraw(600);  // уйдёт в минус на 100, но лимит позволяет

    sav.display();  // [Сберегательный] Владелец: Alice, баланс: 1200, ставка: 5%
    chk.display();  // [Чековый] Владелец: Bob, баланс: -100, лимит овердрафта: 200

    return 0;
}
```

Пример такой реализации классов содержит важные моменты:

- Конструктор производного класса обязан вызвать конструктор базового, используя список инициализации;

```cpp
: Account(owner, initial)).
```

- Методы, которые планируется переопределять в наследниках, помечены в базовом классе как virtual. Ключевое слово override в наследнике не обязательно, но крайне рекомендуется;

- Для вызова базовой версии переопределённого метода используется квалифицированное имя *Account::display()*;

## Модификатор доступа protected

В предыдущем примере поля *m_owner* и *m_balance* в классе ***Account*** были объявлены в секции ***protected***. Это третий модификатор доступа наряду с *private* и *public*.

Использование ***protected*** даёт производным классам прямой доступ к внутренним полям, что иногда упрощает код. Например, в *SavingsAccount::withdraw* мы напрямую изменили *m_balance*, унаследованный от ***Account***. Если бы поле было *private*, потребовались бы публичные или защищённые методы доступа.

Однако делать поля *protected* следует с осторожностью. Открывая их наследникам, мы увеличиваем связанность кода: изменение внутреннего представления базового класса может сломать наследников. Более безопасный подход — хранить поля в *private*, а для наследников предоставлять *protected*-методы для контролируемого доступа (например, *get_balance()* и *set_balance()*).

## Рекомендации по оформлению кода

В ситуациях, когда пишется код с наследованием классов, следует придерживаться следующих рекомендаций:

- ***Деструктор базового класса должен быть виртуальным.*** Если предполагается, что объекты будут удаляться через указатель на базовый класс, виртуальный деструктор гарантирует корректный вызов деструктора наследника. Даже если прямо сейчас такой потребности нет, лучше сразу объявить virtual ~Base() = default;

- ***Используйте override.*** Ключевое слово ***override*** после сигнатуры метода в наследнике явно говорит компилятору, что вы намерены переопределить виртуальную функцию. Если сигнатура не совпадёт с базовой, компилятор выдаст ошибку, предотвращая скрытые баги;

- ***Не наследуйте только ради повторного использования кода.*** Наследование оправдано только при наличии отношения "является". Если классу просто нужна чужая функциональность, лучше применить композицию. Неправильное наследование запутывает архитектуру и влечёт проблемы при будущей поддержке кода;

- ***Избегайте глубоких иерархий.*** Цепочки из 4–5 и более уровней наследования трудно воспринимать и поддерживать. Старайтесь, чтобы иерархия была плоской. Если глубина растёт, возможно, система спроектирована недостаточно гибко.

## Композиция как альтернатива наследованию

Часто начинающие разработчики, узнав о наследовании, начинают строить иерархии там, где они не нужны. Классический антипример – класс ***Car***, наследующий от ***Engine*** (Автомобиль наследует Двигатель). Это нарушает отношение "является": автомобиль не является двигателем, он содержит двигатель. Правильным решением будет композиция (или агрегация) – включение объекта одного класса в другой в качестве поля.

```cpp
class Engine {
    public:
        void start() { /* ... */ }
        void stop()  { /* ... */ }
};

class Car {
    public:
        Car() : m_engine() {}
        void turn_on()  { m_engine.start(); }
        void turn_off() { m_engine.stop();  }
    private:
        Engine m_engine;   // Композиция
};
```

Есть такэе несколько причин, по которым композиция объектов предпочтительней наследования:

- ***Отсутствие проблем множественного наследования.*** В C++ множественное наследование классов допустимо, но создаёт множество сложностей. Во многих современных языках программирования (например C#, Java) множественное наследование классов запрещено вовсе. Композиция же позволяет использовать функциональность скольких угодно компонентов без ограничений;

- ***Гибкость.*** Композиция позволяет легко заменить используемый объект на другой (например, подставить ***ElectricEngine*** вместо ***Engine***) во время выполнения, что невозможно при жёстком наследовании.
